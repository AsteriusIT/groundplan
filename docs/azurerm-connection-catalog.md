# azurerm connection catalog

How the azurerm Terraform provider expresses a link between two resources, and
what groundplan's graph should do with each kind. Generated by analysing the
**full provider schema** (`terraform providers schema -json`), provider
**hashicorp/azurerm v4.81.0** (1,141 resources), on 2026-07-18.

To regenerate: init an empty Terraform dir pinning `azurerm >= 4.0`, run
`terraform providers schema -json > schema.json`, then scan every resource's
configurable attributes for `*_id` / `*_ids` names and every resource whose
required attributes are (almost) exclusively such references.

## The three connection mechanisms

azurerm has exactly three ways to say "A is connected to B". A correct diagram
has to understand all three, because the same logical link is often writable in
more than one of them.

### 1. Inline reference attributes

A resource carries an `*_id` / `*_ids` attribute (often nested in a block:
`ip_configuration.subnet_id`). The schema has **2,393 configurable reference
attributes** across the provider; **98 resource types point at a subnet**
through 21 different attribute names (`subnet_id`, `virtual_network_subnet_id`,
`delegated_subnet_id`, `pod_subnet_id`, `infrastructure_subnet_id`, …).

groundplan already handles these *generically*: `graph/dependency-edges.ts`
turns every expression reference into a `depends_on` edge, and the generic
containment rule ("anything that references a subnet lands in that subnet",
`graph/containment.ts`) matches by the **resolved node's type**, not the
attribute name — so all 21 spellings of "my subnet" work today without a
catalog.

### 2. Association / join resources

A third resource exists solely to bind two others. This is the mechanism the
diagram currently misses (the reported case: a NAT gateway is bound to its
subnet **only** by `azurerm_subnet_nat_gateway_association` — neither end
references the other, so no inline rule can ever place it).

The provider has **28 `*_association` resources**, plus attachment / binding /
connection / link / peering families (97 resources match a join-name pattern;
219 candidates including structurally-detected ones). The curated,
diagram-relevant set is the catalog below.

Because every resource in a plan becomes a node, a join resource today shows up
as its **own box** with edges to both ends (so nothing is lost — it is just
drawn as plumbing instead of as the relationship it states).

### 3. Inline ↔ association duality

Some links can be written **either** way; the semantic model must normalise both
spellings to the same link or a repo's style dictates its diagram:

| Link | Association resource | Inline alternative |
| --- | --- | --- |
| subnet ↔ NSG | `azurerm_subnet_network_security_group_association` | `azurerm_virtual_network.subnet[].security_group` (inline-subnet style) |
| subnet ↔ route table | `azurerm_subnet_route_table_association` | `azurerm_virtual_network.subnet[].route_table_id` |
| subnet ↔ NAT gateway | `azurerm_subnet_nat_gateway_association` | **none — association only** |
| NAT gw ↔ public IP / prefix | `azurerm_nat_gateway_public_ip_association` / `..._prefix_association` | **none** (`azurerm_nat_gateway` has zero configurable ref attrs) |
| NIC ↔ NSG | `azurerm_network_interface_security_group_association` | none on the NIC; VMSS has `network_interface.network_security_group_id` |
| NIC ↔ LB pool / AppGW pool / ASG / LB NAT rule | the four `azurerm_network_interface_*_association` resources | VMSS inline: `network_interface.ip_configuration.{load_balancer_backend_address_pool_ids, application_gateway_backend_address_pool_ids, application_security_group_ids, load_balancer_inbound_nat_rules_ids}` |
| App Service ↔ subnet (vnet integration) | `azurerm_app_service_virtual_network_swift_connection` (+ `_slot_` variant) | `virtual_network_subnet_id` on `azurerm_linux_web_app` & friends (modern style) |

## Curated topology catalog

`*` = required attribute. "Handled" points at the code that does it today.

### A. Subnet attachments → render as subnet-header chips (GP-89 pattern)

| Join resource | Endpoints | Status |
| --- | --- | --- |
| `azurerm_subnet_network_security_group_association` | `network_security_group_id`* ↔ `subnet_id`* | ✅ handled (`plan-parser.ts` / `hcl-parser.ts` → `associated_ids`) |
| `azurerm_subnet_route_table_association` | `route_table_id`* ↔ `subnet_id`* | ✅ handled (same path, GP-89) |
| `azurerm_subnet_nat_gateway_association` | `nat_gateway_id`* ↔ `subnet_id`* | ❌ **missing — the reported gap** |
| `azurerm_application_load_balancer_subnet_association` | `application_load_balancer_id`* ↔ `subnet_id`* | ❌ missing (App Gateway for Containers) |

The inline-vnet duality (§3 row 1–2) is **also unhandled**: a repo that writes
inline subnets gets no chips at all — `associated_ids` is only ever extracted
from association resources.

### B. NIC attachments → attach to the NIC (which stacks under its VM, GP-86)

| Join resource | Endpoints | Status |
| --- | --- | --- |

| `azurerm_network_interface_application_gateway_backend_address_pool_association` | `backend_address_pool_id`* (→ an **inline** AppGW pool — the expression resolves to the `azurerm_application_gateway` node) ↔ `network_interface_id`* | ❌ missing |
| `azurerm_network_interface_nat_rule_association` | `nat_rule_id`* (→ `azurerm_lb_nat_rule`) ↔ `network_interface_id`* | ❌ missing |

### C. Host stacking → satellite nests under its host (GP-86/GP-88 pattern)

| Join resource | Endpoints | Status |
| --- | --- | --- |

| `azurerm_nat_gateway_public_ip_prefix_association` | `public_ip_prefix_id`* → `nat_gateway_id`* | ❌ missing (the rule matches only `azurerm_public_ip`, not `azurerm_public_ip_prefix`) |

### D. Network edges → collapse the join node into a labelled edge

These joins *are* the relationship; drawing them as boxes is plumbing noise.
Recommended rendering: a direct edge between the two endpoints (labelled with
the join type), the join node hidden — the GP-88 "edges around stacks" spirit.

| Join resource | Endpoints |
| --- | --- |
| `azurerm_virtual_network_peering` | local vnet (by `virtual_network_name`) ↔ `remote_virtual_network_id`* |
| `azurerm_databricks_virtual_network_peering` | `workspace_id`* ↔ `remote_virtual_network_id`* |
| `azurerm_virtual_hub_connection` | `virtual_hub_id`* ↔ `remote_virtual_network_id`* |
| `azurerm_virtual_network_gateway_connection` | `virtual_network_gateway_id`* ↔ one of `local_network_gateway_id` \| `express_route_circuit_id` \| `peer_virtual_network_gateway_id` |
| `azurerm_vpn_gateway_connection` | `vpn_gateway_id`* ↔ `remote_vpn_site_id`* |
| `azurerm_express_route_connection` | `express_route_gateway_id`* ↔ `express_route_circuit_peering_id`* |
| `azurerm_express_route_circuit_connection` | two circuit peerings (cross-connect) |
| `azurerm_private_dns_zone_virtual_network_link` | private DNS zone (by `private_dns_zone_name`) ↔ `virtual_network_id`* |
| `azurerm_private_dns_resolver_virtual_network_link` | `dns_forwarding_ruleset_id`* ↔ `virtual_network_id`* |

### E. Cross-service joins → keep as ordinary edges (already fine generically)

The generic reference edges already draw these correctly; at most the join node
deserves de-emphasis, never chips or stacking: the three
`azurerm_spring_cloud_app_*_association` resources (cosmos / mysql / redis),
the four `azurerm_kusto_*_data_connection` resources,
`azurerm_monitor_data_collection_rule_association` (DCR/DCE → any
`target_resource_id`*), `azurerm_network_security_perimeter_association`
(NSP profile → any resource), `azurerm_communication_service_email_domain_association`,
`azurerm_kubernetes_fleet_member`, the site-recovery / recovery-vault policy
associations, the two virtual-desktop associations, the Palo Alto certificate
associations, `azurerm_vmware_netapp_volume_attachment`, the App Service
certificate / hostname bindings, `azurerm_storage_object_replication`.

### F. Governance / IAM joins → IAM view, not the network canvas

`azurerm_role_assignment` (✅ handled, GP-47), the PIM active/eligible role
assignments, the five `*_policy_assignment` variants,
`azurerm_management_group_subscription_association`,
`azurerm_cosmosdb_sql_role_assignment`, the Kusto principal assignments,
`azurerm_synapse_role_assignment`, MHSM / marketplace role assignments,
`azurerm_lighthouse_assignment`, Redis access-policy assignments,
`azurerm_kubernetes_cluster_trusted_access_role_binding`.

### G. Settings-shaped joins → do nothing special

~120 structurally join-like resources are really *settings of one parent*
(customer-managed-key configs, Sentinel data connectors, AAD administrators,
TDE, diagnostic settings, maintenance assignments, …). The generic reference
edge is the right rendering; never collapse or stack these.

## Gap analysis (code pointers)

- `graph/dependency-edges.ts` — nothing to change: join resources already edge
  to both endpoints, so no link is *lost*, only *mis-drawn*.
- `graph/containment.ts` — the `_association` pass-through exists **only inside
  the `direction: "up"` path** (built for public IPs, GP-86). NAT gateways have
  no rule at all, so they float loose in the diagram; `azurerm_public_ip_prefix`,
  data disks, and private endpoints' ASGs are similarly uncovered.
- `graph/plan-parser.ts` + `graph/hcl-parser.ts` — `associated_ids` extraction
  matches two hard-coded substrings (`_route_table_association`,
  `_security_group_association`) against an owner set of exactly
  `{azurerm_network_security_group, azurerm_route_table}` and a target set of
  exactly `{azurerm_subnet, azurerm_network_interface}`. NAT gateway and ALB
  associations fall through all three filters.
- The **inline duality** (§3) is entirely unhandled in both parsers — inline
  vnet-subnet style repos and VMSS inline pools produce no chips/attachments.
- Frontend (`network-container-node.tsx`, `graph-canvas.tsx`) renders whatever
  `associated_ids` / `parent_id` say — fixes are backend-only.

## Implementation (landed)

The recommendation below was implemented as **one data-driven join table**:
`apps/backend/src/graph/azurerm-joins.ts` (the source of truth from here on —
this document and `azurerm-connection-catalog.json` are the analysis record).

- `classifyJoins` resolves every join resource's references through the table;
  `joinEffects` folds the links into three outputs both producers apply:
  `associated_ids` (attach), `parent_id` via `deriveContainment` (contain), and
  direct inferred `depends_on` edges (edge — deduped, in either direction,
  against the reference edges).
- **Semantics as landed** (two deliberate deviations from the tables below):
  the NAT gateway and the container-ALB **nest in their subnet** (`contain`)
  rather than becoming header chips — a chip cannot carry the NAT gateway's own
  public-IP stack. The private-endpoint ASG is an `attach`. An attach to a
  non-subnet anchor also emits a direct edge (there is no chip band for it to
  ride on).
- Ambiguous containment (a NAT gateway bound to two subnets) degrades to plain
  edges — never a guessed parent. Unknown `*_association` types resolving to
  exactly two nodes fall back to an `edge`, so provider upgrades degrade
  gracefully.
- The VMSS inline NSG/ASG duality is handled (`inlineScaleSetLinks`, both
  producers); web-app `virtual_network_subnet_id` was already covered by generic
  subnet containment. **Deferred:** the vnet inline `subnet[]` style — honouring
  its `security_group` / `route_table_id` needs synthetic subnet nodes, which
  touches node identity and diffing, and deserves its own story.
- Frontend: the network view drops the `edge`-join boxes (the
  `EDGE_JOIN_TYPES` mirror in `lib/graph-layout.ts`) and draws the direct edge;
  chips/stacks needed no changes — they were already generic over
  `associated_ids` / `parent_id`.

## Recommendation (original)

Replace the substring special cases with **one data-driven join table** (the
machine-readable version of this catalog sits next to this file:
`azurerm-connection-catalog.json`): for each join type, its two endpoint
attributes/types and the semantic (`attach` | `contain` | `edge` | `iam` |
`none`). Both parsers and `containment.ts` read the same table; a generic
fallback collapses any unknown `*_association` with exactly two resolved
references into a plain edge, so new provider versions degrade gracefully
instead of drawing plumbing boxes.

Priority order for closing the gap (all landed except inline vnet subnets):

1. `azurerm_subnet_nat_gateway_association` → the NAT gateway placed on its
   subnet (the reported bug). ✅
2. The four remaining NIC associations (B) + public-IP-prefix and data-disk
   stacking (C). ✅
3. Edge-collapse for the peering/connection/link family (D). ✅
4. The inline dualities (§3) — VMSS inline guards ✅, web-app
   `virtual_network_subnet_id` ✅ (was already generic), vnet inline subnets
   **deferred** (synthetic subnet nodes — own story).
