# @groundplan/keycloak-theme

The groundplan **Keycloak theme** — the login, account and email pages a user
sees during OIDC sign-in, restyled to match the app's **carbon** palette
(graphite-black, `#4c8dff` primary, Space Grotesk / Inter / IBM Plex Mono).

Built with [Keycloakify](https://keycloakify.dev) 11 (Vite + React 19 +
Tailwind v4 + shadcn/ui) — the same stack as `apps/frontend`, so the visual
language is literally the same code. It compiles to a Keycloak **provider jar**.

## What's themed

| Theme type | How | Pages |
| --- | --- | --- |
| **login** | custom `Template` + `Login`, a typed `classes` map for the rest | login, register, reset password, OTP, verify-email, update-password, error, info, … |
| **account** | Multi-Page (account-v1), custom `Template` + scoped CSS | personal info, password, authenticator, sessions, applications |
| **email** | native FreeMarker, owned `template.ftl` wrapper | verify email, password reset, executeActions, … |

Colour truth lives in [`src/index.css`](src/index.css) — the app's carbon tokens
flattened into a single `:root` and mapped via `@theme inline`, so every shadcn
utility resolves from the carbon palette. Components never hardcode a colour; a
`design-tokens.test.ts` guard enforces it. Emails are the exception (clients strip
CSS variables + web fonts), so `template.ftl` inlines the palette as literal hex.

## Commands

Run from this directory (or `pnpm --filter @groundplan/keycloak-theme <script>`):

| Command | What it does |
| --- | --- |
| `pnpm dev` | Vite preview of a login page (`?page=register.ftl`, `?page=error.ftl`, …) — no Keycloak needed |
| `pnpm build-keycloak-theme` | Build the provider jar → `dist_keycloak/keycloak-theme-groundplan.jar` |
| `pnpm test` | vitest (design-tokens guard + primitive smoke tests) |
| `pnpm typecheck` | `tsc --noEmit` |
| `pnpm update-kc-gen` | Regenerate `src/kc.gen.tsx` after adding a theme type |

From the repo root: **`pnpm keycloak:build`** builds the jar.

## Local dev / verification

The jar is mounted into the dockerized Keycloak and selected by the realm import
(`loginTheme` / `accountTheme` / `emailTheme` = `groundplan`). Build it **before**
starting Keycloak — providers load at startup:

```bash
pnpm keycloak:build                 # from repo root
docker compose --profile auth up -d # Keycloak :8085 mounts the jar
```

Then, signed in as the dev user (`dev` / `dev`):

- **Login** — visit the app (or the realm's auth endpoint); you land on the carbon login.
- **Account** — `http://localhost:8085/realms/groundplan/account`.
- **Email** — needs an SMTP server; point the realm at one (e.g. MailHog) and
  trigger a verify/reset email. SMTP is intentionally left unconfigured in the
  realm import — it's a per-deployment concern.

## Notes for maintainers

- **Owned vs synced files.** The email templates come from the
  `@keycloakify/email-native` extension and are regenerated by the
  `sync-extensions` postinstall (git-ignored via `src/.gitignore`). Only files we
  customize are "owned" and tracked — take ownership with
  `npx keycloakify own --path "email/html/<file>.ftl"` before editing.
- **Account = Multi-Page (account-v1).** Keycloak 26.0 serves it via the
  `AccountFormService`; the jar targets the `26.0-to-26.1` version range (see
  `vite.config.ts`). The dev realm grants the dev user the `account` client roles
  (`view-profile`, `manage-account`) so the console is reachable.
- **`doUseDefaultCss: false`** everywhere — Tailwind owns the styling; Keycloak's
  stock CSS is not loaded.
